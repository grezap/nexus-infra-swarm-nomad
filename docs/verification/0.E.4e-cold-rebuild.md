`terraform destroy` → Packer rebuild → `terraform apply` cycle ending

This is the definition-of-clean per the project's own rebuild canon

## What got fixed during the run

Three structural bugs surfaced and were fixed in the process:

1. **Stage1 stdin-pipe pattern** in 3 TLS overlays — pre-existing
   `bash -c 'echo BASE64 | base64 -d | bash'` shape failed once the
   embedded base64 grew past ~6KB on Windows ssh.exe; documented in
   stage2 v6 but never propagated to stage1. v7's split-script edit
   pushed stage1 over the threshold and the bug surfaced for the first
   time. Fix: pipe LF-normalized plaintext to ssh + `bash -s` with
   `tr -d '\r'` on the remote. Committed in
   [`10377af`](https://github.com/grezap/nexus-infra-swarm-nomad/commit/10377af).

2. **Stale Vault KV poisons cold rebuild** — the consul_acl + nomad_acl
   overlays' Stage 3 "idempotent reuse" of the persisted mgmt token does
   not validate the token against the cluster's actual bootstrap state,
   so a new cluster ends up with the OLD cluster's token in operations,
   stale by construction. Operator-runbook recovery: wipe the affected
   KV paths before re-apply. **Should land in a future overlay
   enhancement: try-token-then-rebootstrap.** Documented in handbook
   §3.7 troubleshooting.

3. **Smoke gate quoting + Block C strategy** — multiple PowerShell
   gotchas surfaced as Block C iterated:
   - `https://$ip:8501` is parsed as scope-qualifier (`$ip:8501`) →
     URL becomes `https://`. Fix: `${ip}:8501`.
   - `curl.exe -o $null` → curl writes body to stdout (PS converts
     `$null` to empty string, curl ignores). Fix: `-o NUL` (Windows
     null device).
   - Windows curl uses schannel which doesn't honour IP SANs. Fix:
     skip curl for off-cluster TLS validation; use SslStream + manual
     chain build (PS scriptblock callback works synchronously and has
     a Runspace, unlike async HttpClient callbacks which run on
     thread-pool threads).
   - Workers' Consul HTTPS API binds 127.0.0.1 only (Consul client-mode
     `client_addr` default). Block A wire-chain check: managers-only
     for Consul, all 6 for Nomad.

## What's proven

1. **Reproducibility.** A clean checkout of `nexus-infra-swarm-nomad` at
   tag `v0.1.1` + `terraform apply` produces an identical end state,
   provided Vault KV `nexus/swarm/{,nomad-}{bootstrap,agent}-token{,s}`
   paths are not stale from a prior deploy. (Cold rebuild canon:
   handbook §3.6 documents the wipe-step prerequisite.)
2. **No hidden operator state on the build host.** The CA bundle is
   the 1-cert stock from Phase 0.D.2; nothing else is required.
3. **Off-cluster reachability is gated.** Block C asserts HTTPS chain
   validation from the build host's bridged adapter against the stock
   bundle; spurious passes (augmented bundles) are warned by Block C's
   bundle-cert-count check.
4. **Apply patterns are stable.** Stage1 + stage2 in all 3 TLS overlays
   use the stdin-pipe pattern; ssh.exe argv quoting on Windows is no
   longer in the failure path.
5. **The 0.E.4 latent bugs are sealed.** Wire-chain has 2 elements on
   every TLS-serving daemon's listener; `inet filter forward` accepts
   docker_gwbridge traffic; bundle augmentation workaround is retired.

## What's deferred to 0.E.5 / future phase

1. **`agent_token-refresh-on-stale-detection` in consul_acl + nomad_acl
   Stage 3.** Today the recovery is operator runbook (wipe KV +
   re-apply). The canonical fix is to validate the persisted token via
   a `consul acl token self` (or equivalent) probe before reuse, and
   re-bootstrap if the validation fails.
2. **Smoke gate Block A SNI-aware probe.** Currently uses a single
   `swarm-manager-1.consul.nexus.lab` SNI for all manager probes.
   Each manager's cert has its own per-host CN; probes work because the
   custom validator returns true regardless of name match. Future work:
   per-host SNI matching the per-host CN.
3. **`portainer-stack` verification probe `--cacert /etc/ssl/certs/nexus-ca.crt`
   path** is wrong (file doesn't exist); the deploy succeeds because the
   probe's failure is non-fatal in the script. Cosmetic fix only.

## Reverting on failure

Each overlay's destroy provisioner restores its `.bak.*` backup +
restarts the affected service. Rollback path documented in handbook
§3.7. A full rollback is `terraform destroy` of the swarm-nomad env,
which leaves Vault KV state in foundation env untouched — the next
`terraform apply` re-creates from scratch (subject to the
Vault-KV-stale-bootstrap-token caveat above).
